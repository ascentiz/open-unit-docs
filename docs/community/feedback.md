# Feedback

Feedback is most useful when it is structured and actionable.

## Issue Types

Use the following categories when opening feedback:

| Type | Include |
| --- | --- |
| Documentation gap | Page URL, missing detail, and the task you were trying to complete |
| API feedback | Current contract, expected behavior, and integration constraint |
| Bug report | Reproduction steps, logs, versions, and observed impact |
| Safety concern | Exact condition, affected workflow, and immediate mitigation taken |
| Research inquiry | Study objective, technical needs, and expected timeline |

## Recommended Workflow

1. Search for an existing issue or discussion.
2. Open a new issue with logs, screenshots, or traces where useful.
3. Mark safety-relevant issues clearly and avoid burying them in general discussion threads.
4. For research collaboration requests, include your lab context and experiment goals up front.

## Response Expectations

As an early-stage platform, response times may vary by topic:

- documentation fixes are typically straightforward
- API or architecture requests may require product and safety review
- research requests usually benefit from a short technical summary rather than a general inquiry

## Minimum Useful Bug Report

Include:

- Open Unit software version
- Exoskeleton Unit firmware version
- exact command or workflow attempted
- observed state and fault outputs
- whether the behavior is repeatable
