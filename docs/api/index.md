# API Overview

The Open Unit API surface is expected to grow over time, but it should remain organized around a few stable concepts:

- **session management** for requesting and tracking command authority
- **control interfaces** for bounded command submission
- **telemetry interfaces** for device state and experiment data
- **system interfaces** for health, faults, and configuration inspection

## Current Documentation Philosophy

This section uses realistic placeholders rather than pretending a final reference already exists.

That is intentional. Early platform teams benefit more from clear contracts and naming discipline than from speculative autogenerated references.

## Expected API Characteristics

The eventual API should provide:

- versioned schemas
- machine-readable error responses
- explicit units for numeric values
- transport-neutral concepts that can be mapped to gRPC, REST, ROS 2, or a device-local RPC layer

## API Maturity Model

| Stage | Expectation |
| --- | --- |
| Preview | Concepts and message shapes may change quickly |
| Beta | Backward compatibility targets begin to matter |
| Stable | Versioning and deprecation policy are enforced |
